# Security Policy

This repository is primarily a research documentation repository, not an online service or application-code repository.

Even so, some issues should still be handled privately, for example:

- accidental inclusion of sensitive information
- material that should not be public
- links that point to unsafe content
- security-related wording that may mislead readers

## How To Report

- Please do not open a public issue first for sensitive reports.
- If GitHub private vulnerability reporting is available, use it first.
- If that path is not available, contact the repository owner privately through the GitHub profile.

When reporting, it helps to include:

- the affected file path
- the relevant commit hash or branch
- a short description of the issue
- the impact as you understand it

## Supported Scope

- Security-related issues are handled against the latest state of the `main` branch only.
- Older commits and historical document versions are not maintained separately by default.

## What Does Not Need A Security Report

These are better handled through a normal issue or pull request:

- ordinary documentation polish
- non-sensitive typo or link fixes
- general disagreement about source interpretation
